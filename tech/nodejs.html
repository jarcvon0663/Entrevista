🟢 Node.js Fundamentals (20–30 min)
🔄 The Event Loop
Core Concept

"How does the Event Loop work?"

"Node.js runs JavaScript on a single thread, but the Event Loop allows it to handle thousands of concurrent operations without blocking, by delegating I/O work to the system and processing callbacks when they complete."

Event Loop Phases
// Simplified order of execution
1. Timers        // setTimeout, setInterval callbacks
2. Pending callbacks
3. Poll           // Retrieve new I/O events
4. Check          // setImmediate callbacks
5. Close callbacks

// Microtasks (Promises, process.nextTick) run 
// BETWEEN each phase, before moving to the next
Demonstrate you understand execution order
console.log('1 - sync');

setTimeout(() => console.log('4 - timeout'), 0);

Promise.resolve().then(() => console.log('3 - promise'));

console.log('2 - sync');

// Output: 1 - sync, 2 - sync, 3 - promise, 4 - timeout
// Sync code first, then microtasks (Promises), then macrotasks (timers)
🧵 Process vs Thread

Process: an independent instance of a running program with its own memory space.

Thread: a unit of execution within a process; threads share the same memory.

Node.js runs your JS on a single main thread, but uses a thread pool (libuv) internally for I/O operations like file system and DNS.

🚫 Why Single-Threaded?

"Node.js uses a single thread for JS execution to avoid the complexity of multi-threaded programming (race conditions, locks). It achieves concurrency through the Event Loop and non-blocking I/O, delegating heavy operations to the OS or worker threads."

🛑 What Blocks the Event Loop
Common Pitfall
  • Heavy synchronous computation (large loops, complex regex)
  • Synchronous file system calls (fs.readFileSync)
  • JSON.parse/stringify on very large objects
  • Synchronous crypto operations
  • Blocking loops without yielding control
💡
Solution: Use async versions of APIs, offload CPU-intensive work to Worker Threads, or break large tasks into chunks with setImmediate.

"Variable and function declarations are moved to the top of their scope during compilation. var is hoisted and initialized as undefined; let/const are hoisted but stay in the Temporal Dead Zone until declared."

🔒 Closures
Must Explain Clearly

"A closure is a function that retains access to variables from its outer scope even after that outer function has finished executing."

function counter() {
  let count = 0;
  return () => {
    count++;
    return count;
  };
}

const increment = counter();
increment(); // 1
increment(); // 2 - "count" persists between calls
⚡ Promise.all() vs Promise.allSettled()
Frequently Asked
Promise.all()
try {
  const results = await Promise.all([
    fetchUser(), fetchOrders()
  ]);
} catch (e) {
  // Fails FAST if ANY promise rejects
  // You lose results from the others
}
Promise.allSettled()
const results = await Promise.allSettled([
  fetchUser(), fetchOrders()
]);
// Always resolves, gives you status
// per promise: 'fulfilled' or 'rejected'
results.forEach(r => {
  if (r.status === 'fulfilled') { ... }
});

When to use each: Use all() when all results are required to proceed. Use allSettled() when you want partial results even if some calls fail.

🌐 REST APIs
🎯 Designing a REST Endpoint
Design Question

Be ready to design a resource on the spot: naming, HTTP verbs, status codes, and error format.

Example: Orders Resource
GET    /api/v1/orders          // list (with pagination)
GET    /api/v1/orders/:id      // get one
POST   /api/v1/orders          // create
PUT    /api/v1/orders/:id      // full replace
PATCH  /api/v1/orders/:id      // partial update
DELETE /api/v1/orders/:id      // delete
PUT

Replaces the entire resource. Idempotent — sending it twice has the same effect as once. Client sends the full object.

PATCH

Applies a partial update — only the fields sent. Not always idempotent depending on implementation.

🔐 JWT, OAuth & Refresh Tokens
Security
  • JWT: self-contained token with signed payload; stateless auth
  • Access Token: short-lived (e.g. 15 min), used for API calls
  • Refresh Token: long-lived, stored securely (httpOnly cookie), used to get new access tokens
  • OAuth: delegated authorization framework (login with Google, etc.)
Error handling pattern
app.use((err, req, res, next) => {
  const status = err.statusCode || 500;
  res.status(status).json({
    error: {
      message: err.message,
      code: err.code || 'INTERNAL_ERROR'
    }
  });
});
🚂 Express
🔗 Middleware
Core Concept

"Middleware functions have access to req, res, and next. They execute in order and can modify the request, end the response, or call next() to continue the chain."

const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization;
  if (!token) return res.status(401).json({error: 'No token'});
  
  try {
    req.user = verifyToken(token);
    next();
  } catch (e) {
    res.status(401).json({error: 'Invalid token'});
  }
};

app.get('/api/profile', authMiddleware, (req, res) => {
  res.json(req.user);
});

🗄️ Databases
SQL
  • Joins: INNER, LEFT, RIGHT, FULL combining tables
  • Indexes: speed up reads, cost on writes
  • Transactions: ACID guarantees, BEGIN/COMMIT/ROLLBACK
  • Isolation Levels: Read Uncommitted → Serializable
MongoDB
  • Aggregations: pipeline stages ($match, $group, $lookup)
  • Indexes: single field, compound, text
  • Embedding: nest related data (fast reads, denormalized)
  • Referencing: store IDs (normalized, more queries)
🤔 Embedding vs Referencing
Design Decision

Embed when: data is always read together, doesn't grow unbounded, and rarely changes independently (e.g. address inside user).

Reference when: data is large, shared across documents, or updated independently (e.g. orders referencing a product catalog).

🚀 Performance
🐌 Optimizing a Slow API
Very Common — Senior
Measure first — profile to find the actual bottleneck (DB query, CPU, network) instead of guessing.
Database — add indexes, avoid N+1 queries, use pagination, cache expensive queries.
Caching — Redis for frequently accessed data, HTTP caching headers.
Async operations — parallelize independent calls with Promise.all instead of sequential awaits.
Offload heavy CPU work — use Worker Threads or a queue (BullMQ) for long-running jobs.
💧 Avoiding Memory Leaks
  • Remove event listeners when no longer needed
  • Clear timers/intervals on cleanup
  • Avoid unbounded caches/arrays that keep growing
  • Watch closures holding references to large objects
  • Be careful with global variables

  • node --inspect + Chrome DevTools
  • Clinic.js (doctor, flame, bubbleprof)
  • 0x for flamegraphs

⚛️ React Basics (20–30 min)
🌳 Virtual DOM & Rendering
Core Concept

"React keeps a lightweight in-memory representation of the UI (Virtual DOM). When state changes, React creates a new virtual tree, diffs it against the previous one (reconciliation), and applies only the minimal changes to the real DOM — this is what makes updates fast."

💡
Hooks let you use state and lifecycle features in function components — introduced to avoid the complexity of class components and enable logic reuse.
🎣 React Hooks
📦 useState
const [count, setCount] = useState(0);

Triggers re-render on update. Updates can be batched.

🔄 useEffect
useEffect(() => {
  // side effect
  return () => {} // cleanup
}, [dep]);

Runs after render; use dependency array to control when.

⚡ useMemo vs useCallback
"When would you use useMemo?"
useMemo
const sorted = useMemo(
  () => expensiveSort(list),
  [list]
);
// Memoizes a COMPUTED VALUE
useCallback
const handleClick = useCallback(
  () => doSomething(id),
  [id]
);
// Memoizes a FUNCTION reference

Answer: "I use useMemo when a computation is expensive and its inputs haven't changed — like sorting or filtering a large list — to avoid recalculating on every render. I use useCallback when passing a function to a memoized child component, to keep its reference stable and avoid unnecessary re-renders."

📌 useRef

Persists a mutable value across renders without triggering a re-render. Common for DOM refs or storing previous values.

🧩 Custom Hooks

Extract reusable stateful logic into a function starting with "use", e.g. useFetch, useDebounce. Shows architectural maturity in interviews.

🖼️ Rendering Strategies
CSR

Client-Side Rendering. HTML is empty initially; JS builds the UI in the browser. Fast subsequent navigation, slower first paint, weaker SEO.

SSR

Server-Side Rendering. HTML generated per request on the server. Better SEO and first paint, higher server load.

SSG

Static Site Generation. HTML pre-built at build time. Fastest delivery (CDN), best for content that doesn't change per-request.

⚡ React Optimization
🔁 A Component Renders 100 Times — How Do You Investigate?
Common Scenario
React DevTools Profiler — record a session and see which renders happen and why ("why did this render" flag).
Check props/state changing unnecessarily — new object/array/function literals created on every render break memoization.
Check context — any component consuming a Context re-renders whenever that context value changes, even if unrelated data changed.
Apply fixes — React.memo for the component, useMemo/useCallback for values/functions passed as props, split context into smaller pieces.
🧠 React.memo
export default React.memo(MyComponent);

Skips re-render if props are shallow-equal to previous render.

📝 Forms & Routing
📋 Formik vs React Hook Form

React Hook Form is generally preferred today — uses uncontrolled inputs by default, so far fewer re-renders and better performance. Formik is more established but re-renders more.

🛣️ Routing
  • React Router: nested routes, dynamic params, loaders
  • Protected Routes: wrapper component checking auth before rendering children, redirecting if not authenticated
Protected Route Pattern
function ProtectedRoute({ children }) {
  const { isAuthenticated } = useAuth();
  if (!isAuthenticated) return <Navigate to="/login" />;
  return children;
}

    🎬 Practical Scenarios
    ⚠️
    This is where candidates are truly differentiated. Structure your answer: diagnose → propose → trade-offs.
    📈 "An API receives 20,000 requests per minute. How would you scale it?"
    • Horizontal scaling with a load balancer across multiple instances
    • Cache frequent reads (Redis) to reduce DB load
    • Add a queue (SQS/BullMQ) for write-heavy or async-safe operations
    • Database read replicas for read-heavy traffic
    • Rate limiting and CDN for static/cacheable content
    • Monitor and auto-scale based on CPU/request metrics
    🔁 "A React component renders 100 times. How would you investigate?"

    See the full breakdown in the React Optimization section above — Profiler → identify cause → React.memo/useMemo/useCallback → split context.

    🐢 "The database takes 8 seconds to respond. What would you do?"
    • Run EXPLAIN/EXPLAIN ANALYZE to see the query plan
    • Check for missing indexes on filtered/joined columns
    • Look for N+1 query patterns from the ORM
    • Check connection pool exhaustion
    • Consider caching the result if data doesn't change often
    • Check if it's a resource issue (CPU/memory on DB server)
    🔐 "Implement authentication for an SPA. What architecture would you use?"
    • JWT access token (short-lived) + refresh token (httpOnly cookie, long-lived)
    • Store access token in memory (not localStorage, to reduce XSS risk)
    • Silent refresh flow to renew access token before expiry
    • Protected routes on the client + auth middleware on the server
    • Consider OAuth/OIDC if integrating with third-party identity providers

    Example — Fetch & Filter Pattern
    function UserList() {
  const [users, setUsers] = useState([]);
  const [query, setQuery] = useState('');

  useEffect(() => {
    fetch('/api/users')
      .then(res => res.json())
      .then(setUsers);
  }, []);

  const filtered = users.filter(u =>
    u.name.toLowerCase().includes(query.toLowerCase())
  );

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <ul>{filtered.map(u => <li key={u.id}>{u.name}</li>)}</ul>
    </>
  );
}
