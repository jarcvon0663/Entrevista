¿Qué sabes sobre transacciones en Mendix?

Fundamentos

Concepto

En Mendix, una transacción es el contexto en el que se agrupan operaciones sobre la base de datos para garantizar atomicidad y consistencia.

Cómo funcionan

  • Cada request (ejecución de un microflujo desde UI o API) se ejecuta dentro de una transacción
  • Todas las operaciones de BD (create, change, delete, commit) ocurren dentro de ese contexto
  • Si ocurre un error no controlado → rollback automático
  • Si termina correctamente → commit automático al finalizar
Flujo de una Transacción
Request Inicia
→
Transacción Abre
→
Operaciones BD
→
Commit/Rollback

Dónde se hacen

  • Se ejecutan en el runtime del servidor, no en el cliente
  • Impactan la base de datos configurada (PostgreSQL, SQL Server, etc.)
  • El desarrollador interactúa con ellas desde microflujos

Cuándo se cierra la transacción

  • Al finalizar el microflujo principal o request
  • Si ocurre una excepción no manejada → se revierte
  • Si todo termina bien → commit automático al final
💡 Punto Clave

Las transacciones garantizan que todos los cambios se apliquen juntos o ninguno se aplique, manteniendo la integridad de los datos.

¿Cuándo ocurre el commit en la base de datos?

Persistencia

El commit ocurre cuando:

  • Se ejecuta explícitamente una actividad Commit Object en el microflujo
  • O al final de la ejecución del microflujo/request si no hubo errores
Mendix Microflow 
// Ejemplo de flujo de commit explícito
1. Create Object (MyEntity)
2. Change Object (set attributes)
3. Commit Object (MyEntity) ← Persistencia a BD
4. Call external API
5. Change Object (update status)
6. End (commit automático final si no hay errores)
⚠️ Importante

Un objeto puede estar en memoria sin persistirse hasta que se haga commit. Si el objeto está marcado como "Commit on change" en la entidad, se puede persistir automáticamente al modificarse.

Estados del Objeto

En Memoria
→
Commit
→
Persistido en BD

Best Practices

Principios Fundamentales

1. Responsabilidad Única

Un microflujo debe hacer una sola cosa bien definida.

✓ Bueno ✗ Evitar

2. Nombrado Claro y Consistente

Usar prefijos como:

  • ACT_ para acciones
  • VAL_ para validaciones
  • SUB_ para submicroflujos
Ejemplos de Nombres 
✓ ACT_CreateOrder
✓ VAL_CheckInventory
✓ SUB_CalculateTax

✗ DoStuff
✗ Process
✗ Handle

3. Evitar Lógica Pesada en UI

Mantener la lógica en microflujos, no en páginas.

4. Minimizar Consultas Innecesarias

Evitar retrieves dentro de loops.

Anti-patrón a Evitar 
// ✗ MAL - Retrieve dentro del loop

CustomerList = RETRIEVE Customers FROM OrderList via Association

Manejar errores con flows alternativos y logs.

6. Dividir Lógica Compleja

Si el flujo crece mucho, separarlo en submicroflujos.

7. Evitar Commits Innecesarios

No hacer commit múltiples veces si no es necesario.

8. Optimizar Performance

  • Evitar loops anidados costosos
  • Usar XPath eficiente
  • Usar índices en atributos filtrados frecuentemente

¿En qué etapa del microflujo debería hacerse el commit?

Timing

Regla General

El commit debe hacerse al final del microflujo, después de:

  • Validaciones
  • Reglas de negocio
  • Cálculos
  • Cambios finales en el objeto
Orden Correcto de Operaciones
1. Validar
→
2. Calcular
→
3. Modificar
→
4. Commit

¿Por qué al final?

  • Evita persistir datos inconsistentes
  • Reduce commits innecesarios
  • Permite rollback completo si algo falla
⚠️ Excepción

Cuando necesitas que el objeto exista en BD antes de continuar, por ejemplo:

  • Integraciones externas que requieren un ID
  • Creación de archivos asociados
  • Lógica que depende de persistencia inmediata
Ejemplo Correcto 
// Microflujo: ACT_ProcessOrder

1. Retrieve Order
2. VAL_CheckInventory (validación)
3. SUB_CalculateTotal (cálculo)
4. Change Order (set status, total)
5. SUB_ApplyDiscount (regla de negocio)
6. Commit Order ← AL FINAL
7. Return Order

¿Por qué deberíamos hacer el commit al final?

Consistencia

Razones Fundamentales

1. Consistencia

Si algo falla antes de terminar la lógica, no se guardan datos incompletos.

2. Atomicidad

Todo se guarda o nada se guarda.

3. Performance

Menos commits = menos roundtrips a base de datos.

4. Menor Riesgo de Datos Corruptos

Guardar parcialmente puede dejar estados inválidos.

Comparación de Estrategias
✗ Commits Múltiples
  • Datos parciales en BD
  • Mayor overhead
  • Rollback complejo
  • Estados inconsistentes
✓ Commit al Final
  • Datos completos
  • Mejor performance
  • Rollback simple
  • Estados consistentes
Escenario de Error 
// ✗ MAL - Commit temprano
1. Create Order
2. Commit Order ← Guardado en BD
3. Calculate Tax
4. ERROR! ← Order queda en BD sin tax calculado

// ✓ BIEN - Commit al final
1. Create Order
2. Calculate Tax
3. Validate Payment
4. ERROR! ← Nada se guardó en BD
5. Commit Order ← Solo si todo ok

Diferencia en el Runtime de Mendix

Avanzado

1. Contexto de Transacción

Cada ejecución server-side (un microflujo síncrono invocado desde la UI, REST, etc.) se ejecuta dentro de una transacción del runtime. Esa transacción es la unidad atómica que terminará en commit o rollback al finalizar la request.

2. ¿Qué hace exactamente la actividad Commit?

  • Fuerza que los cambios de un objeto se sincronicen con la BD dentro de la transacción actual
  • Produce SQL INSERT/UPDATE y asignación de ID si aplica
  • Dispara validaciones y event handlers según las opciones
⚠️ Importante

Commit en Mendix no cierra la transacción global del request. Si después de un commit ocurre un error y la transacción hace rollback, esos cambios también se revertirán porque siguen dentro de la misma transacción de base de datos.

Anatomía de una Transacción
Request Inicia
→
Transacción Abre
→
Commit Activity (flush)
→
Más Lógica
→
Commit Final (BD)

3. ¿Cuándo se hace el commit final?

El commit final de la transacción (la confirmación definitiva en la base de datos) sucede cuando termina la ejecución del request/microflujo y el runtime decide confirmar la transacción. Hasta ese momento todo es recuperable por rollback.

4. Diferencias importantes en el runtime

  • Commit vs flush: la actividad Commit actúa como un flush persistente de objetos dentro del Unit of Work del runtime
  • Microflujos asíncronos: se ejecutan en su propia transacción separada
  • Nanoflows: corren en cliente, no tocan la base de datos del servidor
  • Cache de sesión/transacción: dentro de la misma transacción, los retrieves devuelven las mismas instancias en memoria
  • Optimistic locking: commits pueden fallar por versiones concurrentes
  • Eventos y validaciones: commit puede disparar event handlers
Ejemplo con Rollback 
// Dentro de una transacción:
1. Create Order
2. Commit Order ← Escrito en BD (dentro de transacción)
3. Call External API
4. ERROR en API! ← Excepción no manejada
5. ROLLBACK COMPLETO ← Incluso el commit de paso 2 se revierte

// Resultado: Order no existe en BD porque hubo rollback global

5. Reglas prácticas

  • Haz commit al final salvo que necesites: ID del objeto para integraciones, visibilidad inmediata por otro proceso, o persistencia antes de generar archivos
  • No hagas commits dentro de loops salvo necesidad crítica
  • Maneja excepciones específicas de persistencia (optimistic locking, constraint violations)
  • Usa microflujos asíncronos si quieres que la persistencia ocurra en una transacción independiente
💡 Resumen Clave

La actividad Commit escribe los cambios a la BD dentro de la transacción actual pero no la cierra definitivamente. El runtime cierra (commit definitivo) la transacción al terminar la request. Las transacciones asíncronas y las nanoflows se comportan distinto.

¿Qué es Generalización en Mendix?

Domain Model

Concepto

En Mendix, generalización es la herencia entre entidades del domain model: una entidad (padre) define atributos y asociaciones comunes, y una o varias entidades (hijas) heredan esos elementos y se comportan como subclases del padre.

Ejemplo de Generalización
Persona (Padre)
+ Nombre
+ Edad
+ Email
→
Empleado
+ Salario
+ Cargo
Cliente
+ Puntos
+ Nivel

Qué implica y cómo funciona

  • Reutilización: las entidades hijas reciben atributos y asociaciones del padre, evitando duplicación
  • Polimorfismo: una instancia de la hija también es válida como instancia del padre
  • Modelado: se crea arrastrando la línea de generalización en el domain model
  • Persistencia: las instancias siguen siendo instancias concretas

Casos de uso típicos

  • Entidades que comparten muchos atributos y asociaciones (ej: Persona → Empleado, Cliente)
  • Crear una entidad padre abstracta con comportamiento/atributos comunes
  • Simplificar relaciones: asociar con la entidad padre para relacionarla con cualquier subtipo
⚠️ Limitaciones y Precauciones
  • No abuse de jerarquías profundas; aumenta complejidad
  • Usar generalización solo si hay atributos/relaciones compartidos claros
  • Verifica cómo afectan las reglas de negocio y seguridad
  • Evitar cambios frecuentes en la jerarquía en producción
📌 Respuesta Corta para Entrevista

"Generalización en Mendix es la herencia entre entidades del domain model: el padre define atributos y asociaciones comunes que las entidades hijas heredan. Permite reutilización y polimorfismo, y se usa cuando varias entidades comparten estructura o comportamiento."

¿Cómo se ve la Generalización en la Base de Datos?

Database

Implementación en BD

En Mendix la generalización se materializa con varias tablas relacionadas (table-per-class / TPT-like): la entidad padre y cada entidad hija tienen su propia tabla; las filas usan el mismo ID.

Ejemplo: Persona → Empleado
Tabla: Persona
id (PK) INT
Nombre VARCHAR
Edad INT
FechaCreacion DATETIME
TipoEspecialización VARCHAR
→
Tabla: Empleado
id (PK/FK) INT
Salario DECIMAL
Cargo VARCHAR
FechaContrato DATE

Comportamiento al leer/escribir

  • Al commit: se insertan/actualizan filas en ambas tablas (primero padre, luego hija), manteniendo el mismo identificador
  • Al recuperar: el runtime realiza joins entre las tablas para construir el objeto en memoria
  • Al borrar: se eliminan las filas de la tabla hija y de la tabla padre
SQL Generado (simplificado) 
-- Al crear un Empleado:
INSERT INTO Persona (id, Nombre, Edad, TipoEspecialización) 
VALUES (123, 'Juan Pérez', 30, 'Empleado');

INSERT INTO Empleado (id, Salario, Cargo) 
VALUES (123, 50000, 'Developer');

-- Al recuperar un Empleado:
SELECT p.*, e.* 
FROM Persona p
JOIN Empleado e ON p.id = e.id
WHERE p.id = 123;

Impacto y consideraciones prácticas

  • Performance: cada nivel de herencia añade joins en las consultas
  • Consistencia: el ORM mantiene la integridad (mismo ID en ambas tablas)
  • Jerarquías profundas: penalizan el rendimiento significativamente
⚠️ Alternativas

Asociación one-to-one: Si los datos compartidos son pocos o la especialización afecta mucho la performance, usar entidades separadas ligadas evita joins en cada acceso.

📌 Respuesta Corta para Entrevista

"Mendix guarda cada entidad en su propia tabla. Con generalización hay tabla padre con columna de tipo y tabla(s) hija(s) que comparten el mismo ID; recuperar un objeto especializado implica joins, afectando performance, por eso se usa cuando conviene el polimorfismo."

¿Qué es una Asociación y qué tipos existen?

Relationships

Concepto

En Mendix una asociación une dos entidades del domain model y define cómo sus instancias se relacionan: cardinalidad, obligatoriedad y nombres de rol (navegación) entre ambas entidades.

Tipos Principales

1. Uno a Uno (1 — 1)

  • Una instancia de A se relaciona con a lo sumo una de B y viceversa
  • Persistencia: FK en una de las tablas con restricción de unicidad

2. Uno a Muchos (1 — *)

  • Una instancia de A puede relacionarse con muchas de B; cada B apunta a una única A
  • Persistencia: FK en la tabla del lado "muchos"
  • Es el patrón más usado para jerarquías simples

3. Muchos a Muchos (* — *)

  • Varias instancias de A pueden relacionarse con varias de B
  • Persistencia: Mendix crea una tabla intermedia (join table)
  • Más costosa en joins, usar solo si se necesita
Visualización de Asociaciones
1 — 1
Usuario
↔
Perfil
1 — *
Cliente
↔
Órdenes
* — *
Estudiantes
↔
Cursos

Aspectos complementarios

  • Opcionalidad: 0..1 vs 1 (opcional vs obligatoria)
  • Nombres de rol: cada extremo tiene un nombre para navegación
  • Impacto en performance: múltiples asociaciones o many-to-many aumentan joins
SQL para Asociaciones 
-- 1-* : Cliente → Órdenes
CREATE TABLE Order (
    id INT PRIMARY KEY,
    cliente_id INT FOREIGN KEY REFERENCES Cliente(id),
    ...
);

-- *-* : Estudiante ↔ Curso (tabla intermedia)
CREATE TABLE Estudiante_Curso (
    estudiante_id INT FOREIGN KEY REFERENCES Estudiante(id),
    curso_id INT FOREIGN KEY REFERENCES Curso(id),
    PRIMARY KEY (estudiante_id, curso_id)
);
✅ Buenas Prácticas
  • Modelar según acceso a datos (prefiere 1–* para la mayoría)
  • Evitar asociaciones innecesarias
  • Documentar roles y cardinalidades

REST API en Mendix

Integration

Dos Escenarios

Published REST Service

Expones endpoints (GET, POST, PUT, DELETE) desde tu app.

Consumed REST Service

Tu app llama APIs externas usando Call REST.

Exponer una REST API - Consideraciones

Seguridad

  • Autenticación: Basic, API Key, OAuth2 o usuario/rol de Mendix
  • Autorización: configurar roles y acceso a entidades
  • Nunca exponer entidades sin restringir acceso

Diseño del Endpoint

  • Usar nombres claros y versionado: /api/v1/orders
  • Respetar semántica HTTP (GET, POST, PUT, DELETE)

Manejo de Errores

200 OK 201 Created 400 Bad Request 401 Unauthorized 404 Not Found
Ejemplo de Endpoint 
// GET /api/v1/orders/{id}
// Microflow: API_GetOrder

1. Retrieve Order by ID
2. Validate access permissions
3. If not found → Return 404
4. If no access → Return 403
5. Export Object to JSON
6. Return 200 with Order data

Consumir una REST API - Consideraciones

  • Manejo de errores externos: controlar timeouts y respuestas no 200
  • Mapeo: usar Import Mapping bien definido
  • Transacciones: hacer commit solo después de validar toda la respuesta
📌 Respuesta Corta para Entrevista

"En Mendix una REST API puede publicarse o consumirse. Lo crítico es seguridad, diseño correcto de endpoints, manejo adecuado de errores HTTP, validaciones antes del commit y optimización de performance."

Optimización de Performance

Advanced
🚀 Plan de Acción Sistemático

Ante problemas de performance: medir → diagnosticar → optimizar → validar

1. Triage Rápido

  • Reproducir el problema en testing
  • Recolectar métricas: CPU, memoria JVM, conexiones BD
  • Habilitar LogMinDurationQuery

2. Diagnóstico de Causas Comunes

  • Consultas XPath/OQL subóptimas
  • Microflows por cada row (O(n) operaciones)
  • Missing indexes en columnas filtradas
  • Event handlers pesados
  • Jerarquías de generalización profundas

3. Quick Wins

  • Habilitar paginación y limitar page size
  • Añadir índices en columnas de filtros/joins
  • Evitar microflows por fila
  • Mover lógica pesada a procesos asíncronos
