Azure Support Interview Guide

Complete preparation for your Senior Technical Support Engineer interview — covering VM troubleshooting, platform diagnostics, network analysis, identity & access, and production-ready support strategies for Azure environments.

🧠 Senior Support Engineer Mindset
ℹ️
A Senior Support Engineer doesn't just fix issues — they diagnose systematically, prevent recurrence, and improve systems. Show structured thinking, not trial-and-error.
✅ Senior Behaviors
  • Follow structured troubleshooting flows
  • Isolate issues by layer (infra → network → OS → app)
  • Use diagnostics tools before making changes
  • Document findings and root causes
  • Propose preventive measures
  • Communicate clearly throughout
  • Think about automation and scale
❌ Junior Red Flags
  • Jump to conclusions without investigation
  • "Just restart it" without understanding why
  • Skip network diagnostics
  • Don't check logs or metrics
  • Assume user error without validation
  • Make changes without documenting
  • Don't follow up or prevent recurrence
⚠️
Critical: If you answer "I check credentials and that's it" to VM connectivity issues, you're not demonstrating senior-level thinking. You must show a structured troubleshooting flow covering infrastructure, network, identity, and OS layers.
🖥️ VM Connectivity Troubleshooting
🎯 The Question
Critical Scenario

"A user cannot connect to their Azure VM. How do you troubleshoot this?"

This question tests whether you have a structured approach or just guess randomly. A senior follows a logical flow through all layers.

🚨
Red Flag Answer: "I reset the password and restart the VM." — This shows no diagnostic process and could waste time or make things worse.
🔄 Structured Troubleshooting Flow
Clarify the Problem — Before touching anything, understand the exact situation.

Ask:

  • How is the user trying to connect? (RDP / SSH / Bastion)
  • What's the exact error message?
  • Is this affecting one user or multiple?
  • When did it start? Any recent changes?
  • Was it working before?
Interview Answer
"First, I clarify the issue: how the user is trying to connect 
(RDP/SSH), the exact error message, and whether it's affecting only 
one user or multiple. I also ask when it started and if there were 
any recent changes."
Validate VM Infrastructure Status — Ensure Azure platform and VM are healthy.

Check:

  • VM state: Running / Stopped / Deallocated
  • Health status in Azure Portal
  • Instance view for any platform issues
  • Resource consumption (CPU, Memory, Disk)
  • Recent Azure maintenance or regional issues
Azure CLI
# Check VM status
az vm get-instance-view --name myVM --resource-group myRG \
  --query "instanceView.statuses"

# Check regional health
az vm list-skus --location eastus --output table
Interview Answer
"I check if the VM is running and healthy by reviewing its state, 
health status, and resource consumption in the Azure Portal or via CLI."
Review Network Connectivity — This is where many people fail. Network issues are the most common cause.
⚠️
Critical: Not checking network is a major red flag in senior interviews.

Validate:

  • NSG Rules: Inbound rules allow RDP (3389) or SSH (22)
  • Public IP: Does VM have public IP? Is it assigned correctly?
  • VPN/ExpressRoute: If private access, is connection up?
  • Azure Bastion: If using Bastion, is it deployed and healthy?
  • Load Balancer: If behind LB, check health probes
  • Firewall Rules: Azure Firewall or third-party NVA blocking?
Network Watcher - Test Connectivity
# Test if port 3389 (RDP) is accessible
az network watcher test-connectivity \
  --source-resource myVM \
  --dest-address 10.0.0.4 \
  --dest-port 3389

# View effective NSG rules
az network nic show-effective-nsg \
  --name myNIC --resource-group myRG
Interview Answer
"I validate network connectivity, including NSG rules to ensure ports 
3389 (RDP) or 22 (SSH) are open, verify the VM has a public IP or 
proper private access via VPN/Bastion, and use Network Watcher to 
test connectivity. I also check load balancer health probes if applicable."
Check OS-Level Services — If network is fine, problem might be inside the VM.

Validate:

  • RDP service running (Windows: TermService)
  • SSH daemon running (Linux: sshd)
  • OS firewall not blocking ports internally
  • Listening on correct ports (netstat/ss)
  • System resources not exhausted
Windows - Check RDP Service
# Check if TermService is running
Get-Service TermService

# Check firewall rules
Get-NetFirewallRule -DisplayName "Remote Desktop*"

# Check listening ports
netstat -ano | findstr 3389
Linux - Check SSH
# Check sshd status
sudo systemctl status sshd

# Check if listening on port 22
sudo ss -tlnp | grep 22

# Check firewall
sudo iptables -L -n
Interview Answer
"If the VM is reachable at the network level, I check the OS layer 
to ensure RDP or SSH services are running and not blocked by internal 
firewall rules. I also verify the service is listening on the correct port."
Verify Identity & Permissions — Authentication and authorization issues are common.

Check:

  • User has valid credentials
  • Account not locked or expired
  • Azure AD (Entra ID) integration configured correctly
  • User has permissions to access the VM
  • MFA not blocking access
  • Conditional Access policies not interfering
Interview Answer
"I verify user credentials and permissions, especially if Azure AD 
(Entra ID) integration is involved. I check if the account is locked, 
expired, or if Conditional Access policies might be blocking the connection."
Use Advanced Diagnostics — When the obvious checks don't reveal the issue.

Tools:

  • Boot Diagnostics: Screenshot of VM console
  • Serial Console: Direct access to VM without network
  • System Logs: Windows Event Viewer / Linux syslog
  • Azure Monitor: Metrics and alerts
  • Connection Troubleshoot: Network Watcher diagnostic
Interview Answer
"If needed, I use diagnostics tools like Boot Diagnostics to see the 
console screenshot, Serial Console for direct access, and system logs 
to identify the root cause. Network Watcher's connection troubleshoot 
helps identify network path issues."
Apply Appropriate Fix — Based on findings, implement the targeted solution.

Common Fixes:

  • Reset user password/SSH key
  • Add/modify NSG rule to allow port
  • Restart RDP/SSH service via Serial Console
  • Fix OS firewall configuration
  • Restart the VM if needed
  • Reconfigure network settings
  • Update load balancer health probe
Azure CLI - Reset Password
# Reset VM user password
az vm user update \
  --resource-group myRG \
  --name myVM \
  --username adminuser \
  --password 'NewSecureP@ssw0rd!'
Interview Answer
"Depending on the root cause, I apply the appropriate fix, such as 
resetting credentials, updating NSG rules, restarting services via 
Serial Console, or restarting the VM if necessary."
Communicate & Document — Keep stakeholders informed throughout.
  • Keep user informed of progress
  • Confirm access is restored
  • Document the issue and resolution
  • Create KB article if common issue
  • Update runbooks if needed
Interview Answer
"I keep the user informed throughout the process and confirm access 
is restored. I also document the issue, root cause, and resolution 
for future reference."
Prevention & Improvement — Senior engineers think beyond the immediate fix.

Suggest:

  • Implement monitoring and alerts
  • Improve access controls (Azure Bastion)
  • Automate health checks
  • Review NSG rules regularly
  • Set up Azure Monitor for connectivity
  • Create runbooks for common issues
Interview Answer
"If needed, I suggest improvements like better monitoring, access 
controls such as Azure Bastion, or automation to prevent similar 
issues in the future."
💯 Complete Senior Answer
Reference Template 
"First, I clarify how the user is trying to connect and the exact 
error message. Then, I check whether the VM is running and healthy.

Next, I validate network connectivity, including NSG rules to ensure 
required ports (RDP 3389 / SSH 22) are open, and whether the VM is 
accessible via public IP, VPN, or Bastion.

If the network is fine, I check the OS level, ensuring RDP or SSH 
services are running and not blocked by internal firewall rules.

I also verify user credentials and permissions, especially if Azure 
AD authentication is involved.

If needed, I use diagnostics tools like Boot Diagnostics, Serial 
Console, or logs to identify the issue.

Based on the findings, I apply the appropriate fix, such as resetting 
credentials, updating NSG rules, or restarting services or the VM.

Finally, I confirm access with the user and document the issue. If 
appropriate, I suggest preventive measures like monitoring or improved 
access controls."
🔒 Troubleshooting Without VM Access
ℹ️
Advanced Question: "How do you troubleshoot if you don't have access to the VM?" — This separates those who rely on VM access vs. those who truly understand Azure operations.
🛠️ External Diagnostics Tools
  • Boot Diagnostics: See console screenshot without logging in
  • Serial Console: Direct console access bypassing network
  • Network Watcher: Test connectivity and diagnose network paths
  • Azure Monitor: View metrics (CPU, disk, network) without VM access
  • Activity Log: See recent operations and changes
  • Resource Health: Platform-level health status
🔍 What You Can Check
  • VM state and health status
  • NSG effective rules
  • Public IP configuration
  • Network topology and routing
  • Boot errors via diagnostics screenshot
  • Resource metrics and alerts
  • Recent configuration changes
Azure CLI - Diagnostics Without Access
# Enable boot diagnostics
az vm boot-diagnostics enable --name myVM --resource-group myRG

# Get boot diagnostics screenshot URI
az vm boot-diagnostics get-boot-log --name myVM --resource-group myRG

# Check effective NSG rules (no VM access needed)
az network nic show-effective-nsg --name myNIC --resource-group myRG

# View activity log for recent changes
az monitor activity-log list --resource-group myRG --max-events 50
🎯 Senior Answer
Shows Deep Azure Knowledge

"Without VM access, I can still diagnose most issues using Azure's external tools. I check Boot Diagnostics for console screenshots to see boot errors, use Serial Console for direct access bypassing network issues, and leverage Network Watcher to test connectivity and view effective NSG rules. Azure Monitor provides metrics showing resource consumption, and Activity Log reveals recent changes. This approach allows me to identify whether the issue is platform, network, or OS-level without needing to log into the VM."

🔬 Platform vs Application Issues
⚠️
Critical Skill: If you can't distinguish platform from application issues with clear criteria, you're not ready for senior level. This requires systematic isolation by layers.
🎯 The Approach: Isolate by Layers
Systematic Diagnosis

"I distinguish between platform and application issues by isolating each layer and validating with evidence, not assumptions."

1️⃣ Infrastructure (Azure Platform)
First validate if Azure itself is having issues

Check:

  • VM state (running, restarting, unavailable)
  • Disk and storage accessibility
  • Networking (NSG, Load Balancer, IP)
  • Azure Service Health (regional incidents)
  • Recent platform maintenance

Platform Issue Indicators:

  • VM won't start or fails without recent changes
  • Multiple resources affected in same region
  • Azure alerts about platform degradation
  • Network issues with no config changes
2️⃣ Connectivity & Intermediate Layer
Validate communication between components

Check:

  • Load Balancer health probes
  • Timeouts between services
  • Network Watcher connectivity tests
  • DNS resolution

This could be misconfiguration (not platform) OR actual Azure network issue.

3️⃣ Operating System
If infrastructure is healthy, check the OS

Check:

  • CPU/Memory saturation
  • Services stopped or crashed
  • System logs and errors
  • Disk space exhausted

OS Issue Indicators:

  • VM responds but service doesn't
  • Resource exhaustion
  • OS-level errors in logs
4️⃣ Application
If OS and infrastructure are healthy, it's the app

Check:

  • Application logs and errors
  • HTTP 5xx errors
  • Dependencies (database, APIs, external services)
  • Application-specific metrics

Application Issue Indicators:

  • Errors in application logs
  • Issues started after deployment
  • Only specific endpoints/features failing
  • Infrastructure metrics normal but app returns errors
5️⃣ Correlation (Senior Differentiator)
Correlate metrics, logs, and changes to identify origin

Correlate:

  • Metrics across all layers
  • Recent deployments or changes
  • Timeline of events
  • Scope of impact (one service vs. many)
🔵 Platform Issue Examples
  • Multiple VMs fail simultaneously
  • Azure reports regional incident
  • No recent changes in your environment
  • Infrastructure metrics show degradation
  • Azure Service Health alerts
🟠 Application Issue Examples
  • Started after code deployment
  • Infrastructure healthy but app errors
  • Only specific endpoints affected
  • Application logs show exceptions
  • Dependencies (DB, API) timing out
🎯 Complete Senior Answer
Reference Template 
"I distinguish between platform and application issues by isolating 
each layer systematically.

First, I check Azure infrastructure health, including VM status, 
networking, storage, and any regional issues using Azure Service Health.

If infrastructure is healthy, I validate connectivity between components 
and then check the OS level for resource usage or service failures.

If everything looks stable at the infrastructure and OS levels, I 
analyze application logs and behavior.

I also correlate this with recent changes, such as deployments or 
configuration updates.

For example, if multiple resources fail without changes, it may 
indicate a platform issue. If errors start after a deployment and 
infrastructure is stable, it's likely an application issue.

The key is using evidence from metrics and logs at each layer, not 
making assumptions."
🚨
Red Flags: "Just check the logs" (which logs?), not mentioning infrastructure validation, not discussing correlation, or mixing symptoms with root causes.
📊 Key Metrics for Decision Making
🔍 What Metrics to Check
Data-Driven Diagnosis

"Specific metrics help determine if the issue is platform or application related. Here's what to look for at each layer:"

Layer Key Metrics What It Indicates
Platform VM availability, disk IOPS, network packets dropped Azure infrastructure health
Network Network In/Out, failed connections, latency Connectivity and throughput issues
OS CPU %, memory %, disk queue length, available memory Resource exhaustion at OS level
Application HTTP response codes, request duration, exceptions, dependency failures Application-specific errors
Azure Monitor - Query Metrics
# Get CPU percentage for VM
az monitor metrics list \
  --resource /subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.Compute/virtualMachines/{vm} \
  --metric "Percentage CPU" \
  --start-time 2024-01-20T00:00:00Z

# Get network metrics
az monitor metrics list \
  --resource {resourceId} \
  --metric "Network In Total" "Network Out Total"
💡
Senior insight: "I look at trends over time, not just current values. A spike in errors after a deployment points to application issues. Gradual platform degradation points to infrastructure problems."
💻 Running Bash on Windows
❓ The Question
Common Interview

"Is it possible to run Bash in Windows CMD?"

This tests your understanding of Windows environments and modern development tools.

⚠️
Quick Answer: Not directly in CMD. CMD doesn't execute Bash natively. But there are several valid ways to run Bash on Windows.
✅ WSL (Recommended)

Windows Subsystem for Linux — The modern, official way.

  • Full Linux environment on Windows
  • Real Bash shell (Ubuntu, Debian, etc.)
  • Native performance
  • Integrated with Windows filesystem
PowerShell - Install WSL
# Install WSL (one command)
wsl --install

# Run Bash from CMD/PowerShell
wsl

# Execute specific Bash commands
wsl ls -la
wsl bash script.sh
🔹 Git Bash

Lightweight Bash environment that comes with Git.

  • Lighter than WSL
  • Basic Bash utilities
  • Good for simple scripts
  • Not a full Linux environment
Usage
# After installing Git, use Git Bash terminal
# Or call from CMD via full path
"C:\Program Files\Git\bin\bash.exe" script.sh
⚙️ PowerShell Integration

PowerShell can invoke WSL or run compatible commands.

# From PowerShell, call WSL
wsl bash -c "./deploy.sh"

# Or invoke Azure CLI (works everywhere)
az vm list --output table
🔧 Other Options
  • Cygwin: Unix-like environment (heavier, less common now)
  • MSYS2: Minimal Unix environment
  • Azure Cloud Shell: Browser-based Bash for Azure
🎯 Senior Answer
Complete Response

"CMD doesn't support Bash natively. However, on Windows I typically use WSL (Windows Subsystem for Linux) to run a full Linux environment and execute Bash commands. It's the modern, official approach from Microsoft.

Alternatively, Git Bash provides a lightweight option for basic Bash scripting.

In practice, when working with Azure, I often integrate Bash scripts with Azure CLI via WSL, or run them directly in Azure Cloud Shell for cloud-native operations. For automation, I'd execute scripts through Azure DevOps pipelines or directly on Linux VMs rather than locally on Windows."

🚨
Red Flags: Saying "yes, directly in CMD" (wrong), not knowing WSL exists, confusing PowerShell with Bash, or not mentioning how this applies to real Azure scenarios.

  --source-resource myVM \
  --dest-address 10.0.0.5 --dest-port 443

az monitor activity-log list --resource-group myRG --max-events 20
🛡️ Prevention & Automation
💡
Senior Mindset: Don't just fix issues — prevent them from happening again. Suggest improvements, automation, and better monitoring.
🔒 Access & Security
  • Use Azure Bastion instead of public IPs
  • Implement Just-In-Time VM access
  • Azure AD integration for authentication
  • Conditional Access policies
  • Regular NSG rule audits
📊 Monitoring & Alerts
  • Azure Monitor alerts for VM health
  • Log Analytics for centralized logging
  • Connection Monitor for network paths
  • Service Health alerts for platform issues
  • Automated runbooks for common fixes
⚙️ Automation
  • Auto-remediation with Azure Automation
  • Health check scripts scheduled via Azure Functions
  • Infrastructure as Code (ARM, Bicep, Terraform)
  • Policy enforcement for configurations
  • Runbooks for common troubleshooting steps
📚 Documentation
  • Create KB articles for common issues
  • Document architecture and dependencies
  • Maintain updated runbooks
  • Post-incident reviews and RCAs
  • Share lessons learned with team
💡 Interview Tips & Best Practices
✅ What Makes You Senior
  • Follow structured troubleshooting flows (infra → network → OS → app)
  • Mention specific Azure tools (Network Watcher, Serial Console, Boot Diagnostics)
  • Differentiate RDP vs SSH scenarios
  • Always check NSG rules and network connectivity
  • Use logs and diagnostics before making changes
  • Propose preventive measures and automation
  • Communicate clearly and document findings
  • Think about correlation and evidence, not assumptions
❌ Red Flags to Avoid
  • Jump straight to "reset password and restart"
  • Don't mention network diagnostics (critical miss)
  • Don't differentiate between RDP and SSH
  • Don't mention NSG rules or ports
  • Skip checking logs and metrics
  • No mention of Azure diagnostic tools
  • Assume it's always user error
  • Don't suggest prevention or improvement
  • Can't explain how to diagnose without VM access
🎯 Key Phrases to Use
Sound Like a Senior
  • "I follow a systematic approach, starting with clarifying the issue..."
  • "I isolate the problem by layer: infrastructure, network, OS, then application..."
  • "I use Network Watcher to test connectivity and view effective NSG rules..."
  • "I correlate metrics and logs with recent changes to identify the root cause..."
  • "If I don't have VM access, I can still diagnose using Boot Diagnostics and Serial Console..."
  • "Based on the evidence, I determine if it's a platform issue or application issue..."
  • "I document the resolution and suggest preventive measures like monitoring and automation..."
💡
Pro Tip: Always frame your answers with "First, I...", "Then, I...", "If that doesn't work, I..." — this shows structured thinking. Avoid vague statements like "I would check things" — be specific about what you check and how.
