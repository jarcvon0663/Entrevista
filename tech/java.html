Comparison

⚙️ How the JVM Works
⚠️
This is a very common question. Be ready to explain the JVM flow and its key components clearly.
🔄 JVM Execution Flow
Core Concept

"The JVM (Java Virtual Machine) is the component that executes Java bytecode and allows the same program to run on any operating system without changes."

Compilation — .java code is compiled with javac to bytecode (.class). This bytecode is OS-independent.
Class Loading — The JVM loads necessary classes into memory using different classloaders (Bootstrap, Extension, Application).
Verification — The JVM validates bytecode to prevent unsafe or corrupted code.
Execution — Bytecode is executed in two ways: interpreted (line by line) or JIT compiled (Just-In-Time: compiles critical parts to native code for better performance).
Memory Areas
  • Heap: where objects live (managed by GC)
  • Stack: local variables and method calls (per thread)
  • Metaspace: class metadata
  • PC Register: execution tracking per thread
Garbage Collector

Automatically manages memory by removing unused objects. Algorithms: G1, ZGC, Shenandoah. Reduces memory leaks and simplifies development.

Execution Engine
  • Interpreter: fast startup
  • JIT Compiler: optimizes runtime performance by detecting and optimizing "hot code"
⚡ The Killer Feature of JVM
Must Know

"What is the killer feature of the JVM?"

"The killer feature of the JVM is its ability to combine portability with high performance through the Just-In-Time (JIT) compiler. It allows Java code to be written once, run anywhere, and still achieve near-native performance by optimizing hot code at runtime."

💡
Pro tip: "Unlike traditional compiled languages, the JVM can optimize based on real execution patterns, not assumptions made at compile time." — This shows deep understanding.
🌐 REST API Levels
ℹ️
The question "How many levels of REST do you know?" refers to the Richardson Maturity Model.
📊 Richardson Maturity Model
4 Levels

"REST has 4 levels according to Richardson's maturity model, from level 0 to level 3."

Level 0 – The Swamp of POX

Uses HTTP but without REST principles.

  • Single endpoint
  • Everything is POST
  • Example: /api for everything
Level 1 – Resources

Introduces resources.

  • Separate URLs per entity
  • Example: /users, /orders
  • But still misuses HTTP (everything POST)
Level 2 – HTTP Verbs

Correctly uses HTTP methods.

  • GET, POST, PUT, DELETE
  • Correct status codes (200, 404, 500)
  • ✅ This is REST "done right"
Level 3 – HATEOAS

API guides the client dynamically through links.

// Response includes navigation
{
  "user": "123",
  "links": [
    { "rel": "orders", "href": "/users/123/orders" }
  ]
}
💡
Industry reality: "In practice, almost nobody fully implements HATEOAS; level 2 is the industry standard."
🏗️ Microservices vs Monolith
🤔 Why Microservices? Why not Monolithic?
Architecture Decision

Brief Summary:

"Microservices solve organizational scalability, independent deployment, and resilience at the cost of greater operational complexity. A monolith is simpler and often sufficient; choose based on the real problem, not trends."

✅ Why Choose Microservices
  • Component scaling: scale only the service that needs more resources
  • Independent deployments: teams deploy without coordinating the entire system
  • Fault isolation: a failure in one service doesn't bring down the whole app
  • Heterogeneous technologies: each service can use the best-fit technology
  • Team ownership: self-contained teams with clear ownership
  • Incremental evolution: replace or version parts without rebuilding everything
⚠️ Microservices Costs
  • Operational complexity: needs orchestration, discovery, gateway, distributed logging, tracing
  • Higher latency: RPC/HTTP calls between services add latency
  • Consistency challenges: eventual consistency and distributed transactions (Sagas)
  • Complex testing: integration and E2E require simulated infra
  • Initial cost: infrastructure and DevOps skills increase time and budget
  • Observability required: without metrics/tracing, debugging is impossible
✅ Why Choose a Monolith
  • Simplicity to start: less infra, lower DevOps learning curve
  • Lower internal latency: in-memory calls, predictable performance
  • Simpler development and testing: single deployment, fewer mocks
  • Lower initial cost: ideal for MVPs or small teams
📋 When to Use Each
Decision Framework

Start with Monolith if:

  • You're a startup/MVP
  • Small team (<6-8 developers)
  • Product in exploration phase
  • Predictable load

Consider Microservices if:

  • Multiple teams (>8-10) blocking each other
  • Parts of the domain have very different scaling or availability requirements
  • Need independent deployments for business reasons
  • Have DevOps culture and capacity
💡
Middle ground: Modular Monolith — clear modules in a single codebase with well-defined boundaries; makes it easier to extract services later.
🎯 Domain-Driven Design (DDD)
📚 What is DDD?
Core Concept

"DDD (Domain-Driven Design) is a software design approach where development centers on the business domain. The idea is to model the system based on how the business actually works, using a common language between developers and domain experts."

Main Idea:

"The goal is for the code to directly reflect business rules, not just the technical part."

🗣️ Ubiquitous Language

A shared language between business and development to avoid misunderstandings. Terms used in code match exactly what the business uses.

🔲 Bounded Context

Divides the system into contexts where a model has a specific meaning. Different contexts can have different models for the same concept.

🏛️ Entities & Value Objects

Entities have identity (User with ID). Value Objects are immutable and defined by their values (Money, Address).

📦 Aggregates

Group entities and guarantee consistency through a root. Only the aggregate root can be referenced from outside.

💾 Repositories

Abstract data access. Provide collection-like interface for aggregates, hiding persistence details.

⚙️ Domain Services

Encapsulate business logic that doesn't naturally fit in entities or value objects.

💡
Why use DDD: "It helps manage complex systems, improves communication with business, and produces more maintainable code aligned with actual requirements."
⚠️
Common mistakes to avoid: Don't say "DDD is just using many layers" or "DDD is only architecture" — that shows you don't understand it well.
🔗 Linked Lists
⏱️ Access Complexity
O(n)

"What is the complexity to get an element from a linked list?"

"The complexity to get an element from a linked list is O(n), because you must traverse the list from the start until you find the element."

Why O(n):

  • In a linked list there's no direct access by index
  • Each node points to the next one
  • To reach position k, you traverse k elements
Array / ArrayList
// Direct index access
int value = array[5]; // O(1)

// Fast random access
ArrayList<Integer> list = new ArrayList<>();
int val = list.get(5); // O(1)
LinkedList
// Must traverse nodes
LinkedList<Integer> list = new LinkedList<>();
int val = list.get(5); // O(n)

// Sequential access required
💡
Pro tip: "Although insertion or deletion in a linked list can be O(1) if you already have the node reference, access is still O(n), making it less efficient for frequent reads."
🔄 Cycle Detection in Linked Lists
⚠️
This is a classic interview question. Be ready to explain Floyd's algorithm clearly.
🔍 The Problem
Classic Algorithm

Scenario: "A linked list where a node points to a previous node instead of the next one — for example, node 5 → node 3 instead of 5 → 6. This creates an infinite loop (cycle)."

Problem:

"In that case, there's no end to the list and traversing it can generate an infinite loop."

✅ Floyd's Algorithm (Tortoise and Hare)
Optimal Solution

"To detect this type of cycle, we use Floyd's algorithm with two pointers:"

Slow pointer — advances 1 step at a time
Fast pointer — advances 2 steps at a time
If there's a cycle — they will eventually meet
If no cycle — fast pointer reaches null
Java — Floyd's Cycle Detection
public boolean hasCycle(ListNode head) {
    if (head == null) return false;
    
    ListNode slow = head;
    ListNode fast = head;
    
    while (fast != null && fast.next != null) {
        slow = slow.next;        // Move 1 step
        fast = fast.next.next;   // Move 2 steps
        
        if (slow == fast) {
            return true;  // Cycle detected!
        }
    }
    
    return false;  // No cycle
}

Complexity:

  • Time: O(n)
  • Space: O(1) — only two pointers!
🔄 Alternative: HashSet Approach
Uses More Memory

"You can also use a HashSet to store visited nodes. If you see a node again, there's a cycle. But this uses O(n) space."

Java — HashSet Approach
public boolean hasCycle(ListNode head) {
    Set<ListNode> visited = new HashSet<>();
    ListNode current = head;
    
    while (current != null) {
        if (visited.contains(current)) {
            return true;  // Cycle found
        }
        visited.add(current);
        current = current.next;
    }
    
    return false;
}

Time: O(n), Space: O(n) — Floyd's algorithm is more space-efficient.

📨 Amazon SQS
☁️ What is SQS?
AWS Service

"SQS (Amazon Simple Queue Service) is a fully managed messaging service from AWS that allows you to decouple system components using queues. Producers send messages to the queue and consumers process them asynchronously."

📋 Standard Queue
  • High scalability
  • Best-effort ordering (not guaranteed)
  • At-least-once delivery (possible duplicates)
  • Nearly unlimited throughput
🎯 FIFO Queue
  • Guaranteed order
  • Exactly-once processing
  • No duplicates
  • Limited to 300 TPS (3000 with batching)
⚙️ Key Concepts
Must Know

Visibility TimeoutTime a message is hidden while being processed
Message RetentionHow long a message lives in the queue (default 4 days, max 14)
Dead Letter Queue (DLQ)Messages that fail multiple times are sent here for analysis
Long PollingReduces empty responses and costs by waiting for messages
👥 How Many Consumers Can SQS Have?
Scalability

Answer: "SQS has no fixed limit on consumers. You can have as many as you need."

Standard Queue:

"Allows multiple consumers without practical limit, with high concurrency, but may have duplicate messages and no guaranteed order."

FIFO Queue:

"Also allows multiple consumers, but processing is limited by message group IDs; within the same group, messages are processed sequentially one at a time. Different groups can be processed in parallel."

💡
Key point: "Parallelism in FIFO depends on how many message groups you have."
⚖️ SQS vs Kafka
🔄 Main Difference
Architecture Choice

"The main difference is that SQS is a managed queue service for simple decoupled messaging, while Kafka is an event streaming platform oriented towards event logs, with high retention and replay capability."

Amazon SQS
  • Model: Queue
  • Consumption: Messages consumed and deleted
  • Replay: No native replay
  • Management: Fully managed (no infrastructure)
  • Best for: Async tasks, decoupling microservices
Apache Kafka
  • Model: Distributed log (streaming)
  • Retention: Messages stored for defined time
  • Replay: Can re-read events
  • Management: Requires management (or MSK/Confluent)
  • Best for: Event streaming, analytics, data pipelines

🎯 When to Use Each

Use SQS if:

  • You want simplicity and quick decoupling
  • Task-based processing
  • Don't need replay

Use Kafka if:

  • Event streaming needed
  • Multiple independent consumers
  • Replay and audit capabilities
  • High throughput requirements
💡
Senior answer: "SQS is pull-based and oriented to task processing, while Kafka is log-based and allows multiple independent consumers with offsets, enabling replay and advanced parallel processing."
🔥 Load Testing
📊 How Did You Do Load Testing?
Common Question

"In my projects, I perform load testing by simulating real traffic to validate system behavior under different load levels."

Define scenarios — Based on real usage: critical endpoints, concurrent user peaks, main business flows (login, frequent queries, heavy processes like payments).
Choose tools — Apache JMeter, Gatling, or k6 to generate concurrent load.
Execute progressive tests — Baseline (normal load), Stress testing (until system breaks), Spike testing (sudden peaks), Soak testing (sustained load over time).
Analyze key metrics — Latency (response time), Throughput (requests per second), Error rate, CPU and memory usage.
Optimize bottlenecks — Slow database queries, concurrency issues, lack of caching. Implement solutions like Redis caching, connection pool tuning, or query optimization.
Validate improvements — Repeat tests to confirm optimizations and ensure stability.
🛠️ Tools
  • JMeter: GUI-based, widely used
  • Gatling: Scala-based, great reports
  • k6: JavaScript, developer-friendly
  • Locust: Python-based
📊 Test Types
  • Baseline: Normal expected load
  • Stress: Push until failure
  • Spike: Sudden traffic bursts
  • Soak: Long duration test
📈 Key Metrics
  • Response time (p50, p95, p99)
  • Throughput (RPS)
  • Error rate (%)
  • Resource usage (CPU/Memory)
💡
To stand out: "In a project we reduced latency by X% by optimizing queries and adding caching, after identifying the bottleneck through load tests."
